1. НАЗНАЧЕНИЕ И ОБЩАЯ ХАРАКТЕРИСТИКА СИСТЕМЫ
--------------------------------------------------------------------------------

Веб-приложение предназначено для загрузки табличных данных (CSV, Excel),
их предпросмотра, интерактивного построения диаграмм различных типов в браузере
и (для зарегистрированных пользователей) сохранения построенных графиков
в персональную коллекцию с возможностью последующей загрузки и удаления.

Ключевые возможности:
  • Работа без авторизации: загрузка файла, привязка к сессии браузера,
    построение графика без сохранения в «библиотеку».
  • Регистрация, вход, выход; личный кабинет (логин, аватар, смена пароля).
  • Светлая/тёмная тема интерфейса; для авторизованных — сохранение выбора
    темы в профиле на сервере, для гостей — в localStorage.
  • Типы диаграмм: линейная, столбчатая, круговая, гистограмма, точечная,
    с областями (реализация на сервере через библиотеку Plotly).
  • Настройки оформления: заголовок, подписи осей, цвета, стиль линии,
    число корзин для гистограммы, шрифт, сетка, легенда, фон.
  • Экспорт графика (через Plotly на клиенте) с учётом темы.
  • REST-подобные JSON-эндпоинты для загрузки, построения графика,
    списка сохранённых графиков, получения/удаления графика по id.

Стек технологий (реализация):
  • Сервер: Python, фреймворк Django (в проекте указана версия 5.2 в
    комментариях settings; в requirements — Django >= 4.2).
  • СУБД: MySQL (основная конфигурация в chartHaven/settings.py);
    для автотестов — SQLite в памяти (chartHaven/settings_test.py).
  • Обработка данных: pandas, numpy; чтение Excel — openpyxl (через pandas).
  • Визуализация на сервере: plotly (graph_objects, express); сериализация
    trace/layout в JSON для отрисовки на клиенте.
  • Клиент: HTML/CSS/JavaScript; отрисовка — Plotly.js с CDN;
    анимации — GSAP (CDN).

ВАЖНО для диплома и безопасности: в файле settings.py хранятся SECRET_KEY
и параметры подключения к MySQL. В тексте диплома указывают, что «секретные
ключи и учётные данные задаются в конфигурации и не публикуются»; в репозиторий
для продакшена их выносят в переменные окружения (.env) и не коммитят.

4. МОДЕЛИ ДАННЫХ И БАЗА ДАННЫХ
--------------------------------------------------------------------------------

4.1. Стандартные таблицы Django
При выполнении migrate создаются таблицы подсистемы аутентификации и админки,
в частности (имена по умолчанию):
  • auth_user — пользователи (id, username, password, is_staff, is_active, даты…).
  • auth_group, auth_permission, связи many-to-many — при использовании групп.
  • django_session — сессии (для гостей и авторизованных).
  • django_admin_log, django_content_type и др.

Данные в этих таблицах — учётные записи администраторов и пользователей,
сессионные ключи; наполнение зависит от эксплуатации, не зашито в код.

4.2. Таблица chart_profile (модель Profile)
Связь один-к-одному с auth_user (поле user_id UNIQUE).
Поля:
  • id — первичный ключ BigAutoField.
  • user_id — FK на auth_user, CASCADE при удалении пользователя.
  • avatar — BinaryField, nullable; бинарные данные изображения аватара.
  • updated_at — DateTimeField, auto_now=True.
  • ui_theme — CharField(max_length=8), default 'light'; варианты: light, dark.
  • email_confirmation_code — CharField(6), blank/null (наследие миграции 0002,
    смена e-mail в UI не используется).
  • pending_email — EmailField, blank/null (то же).

Назначение: расширение учётной записи без изменения стандартной модели User.

4.3. Таблица chart_uploadedfile (модель UploadedFile)
Поля:
  • id — PK.
  • user_id — FK на User, nullable; NULL для анонимной загрузки.
  • session_key — CharField(40), nullable, индекс; для гостей — ключ сессии Django.
  • filename — имя файла (до 255 символов).
  • data — BinaryField; полное содержимое файла в БД.
  • uploaded_at — DateTimeField, auto_now_add.

Логика: авторизованный пользователь «владеет» записями по user_id; гость —
по совпадению session_key и user_id IS NULL. Это ограничивает доступ к файлу
тем же браузером/сессией.

4.4. Таблица chart_savedchart (модель SavedChart)
Поля:
  • id — PK.
  • user_id — FK User, CASCADE (график только у зарегистрированного пользователя).
  • uploaded_file_id — FK UploadedFile, CASCADE.
  • chart_type — строка (line, bar, pie, hist, scatter, area).
  • data_json — JSON: массив traces Plotly после to_dict()['data'].
  • layout_json — JSON: объект layout Plotly.
  • preview_html — TextField: HTML предпросмотра df.head() (таблица).
  • columns_json — JSON-массив имён столбцов.
  • settings_json — JSON-объект настроек UI/графика (см. раздел про create_chart).
  • created_at — DateTimeField, auto_now_add.

Связи: один пользователь — много сохранённых графиков; каждый график ссылается
на один загруженный файл (копия данных на момент сохранения остаётся в UploadedFile).

4.5. Эволюция схемы (миграции chart/migrations)
  • 0001_initial — Profile, UploadedFile (user обязателен), SavedChart (базовые JSON).
  • 0002 — поля email_confirmation_code, pending_email в Profile.
  • 0003 — preview_html, columns_json в SavedChart.
  • 0004 — settings_json в SavedChart.
  • 0005 — ui_theme в Profile.
  • 0006 — session_key в UploadedFile; user в UploadedFile стал optional (null=True).

4.6. Индексы и ограничения
  • session_key в UploadedFile с db_index=True для быстрого поиска по сессии гостя.
  • Остальные индексы — по умолчанию Django (PK, FK).

6. ПРЕДСТАВЛЕНИЯ (РЕАЛИЗАЦИЯ ЛОГИКИ СЕРВЕРА)
--------------------------------------------------------------------------------

6.1. views/charts.py
  • ALLOWED_CHART_TYPES — множество допустимых типов графиков.
  • _axes_validation_error(df, chart_type, x_axis, y_axis) — проверка наличия
    колонок в DataFrame; для hist допускается только ось x.
  • index(request) — render chart/index.html.
  • upload_file(request):
      - Только POST; файл из request.FILES['file'].
      - Расширения: .csv, .xlsx, .xls.
      - Чтение в bytes; pandas read_csv / read_excel из BytesIO.
      - Ошибки парсинга — JsonResponse 400 с текстом.
      - Создание UploadedFile: если user.is_authenticated — user=текущий,
        session_key=None; иначе при необходимости request.session.create(),
        user=None, session_key=текущий ключ сессии.
      - Ответ JSON: preview_data (HTML таблицы head), columns, uploadedFileId,
        uploadedFileName.
  • create_chart(request):
      - Тело JSON: chartType, xAxis, yAxis, chartColor, chartColors, chartTitle,
        xLabel, yLabel, lineWidth, lineStyle, bins, uploadedFileId, saveToLibrary,
        appearance (шрифт, цвет текста, фон, сетка, легенда, позиция легенды).
      - Загрузка UploadedFile через get_uploaded_file_for_request.
      - Повторное чтение файла в DataFrame.
      - Построение fig в зависимости от типа:
          line — go.Scatter (линии+маркеры, dash из line_style);
          bar, pie, hist, scatter, area — plotly.express.
      - fig.update_layout для подписей осей; apply_chart_appearance(fig, appearance).
      - Сериализация numpy в JSON-совместимые типы (to_list рекурсивно).
      - settings_json собирается из входных параметров и appearance.
      - Если save_to_library True: требуется аутентификация; проверка уникальности
        «нормализованного» названия графика (_user_has_saved_chart_with_same_title);
        SavedChart.objects.create(...).
      - Ответ: graphData, layout, columns, preview_data, saved.

6.2. views/helpers.py
  • _normalize_saved_chart_title_for_duplicate_check — trim, пустое = «Без названия»,
    сравнение в нижнем регистре (casefold).
  • _user_has_saved_chart_with_same_title(user, chart_title) — обход сохранённых
    графиков пользователя, сравнение заголовков из settings_json.
  • get_uploaded_file_for_request(request, uploaded_file_id):
      - Авторизованный: UploadedFile с pk и user=request.user.
      - Гость: UploadedFile с pk, user__isnull=True, session_key равен текущей сессии.
  • apply_chart_appearance(fig, appearance) — сброс template, шрифт, сетка осей,
    легенда (позиции top/bottom/left/right), paper_bgcolor/plot_bgcolor.

6.3. views/auth_views.py
  • register_view — GET/POST RegisterForm; при успехе login + redirect index.
  • register_check_username — JSON username, ответ state: empty|invalid|taken|ok.
  • register_validate_password — validate_password Django, список сообщений об ошибках.
  • login_view — POST username/password, authenticate, login или сообщение об ошибке.
  • logout_view — logout, redirect index.

6.4. views/profile_views.py (все с @login_required, кроме косвенно)
  • profile_view — POST «save_profile»: UserForm + ProfileForm (аватар read в bytes);
    POST «change_password»: PasswordChangeFormRu + update_session_auth_hash.
  • verify_current_password — JSON password → check_password.
  • profile_check_username — как регистрация, плюс state «current» для своего логина.
  • save_user_ui_theme — JSON theme ∈ {light, dark} → profile.ui_theme.

6.5. views/api.py
  • get_saved_charts — список графиков пользователя с метаданными (id, тип, имя файла,
    даты, chartTitle из settings).
  • get_chart_by_id — GET: полные данные для восстановления на клиенте; fileData в base64;
    DELETE: удаление записи SavedChart (только своей).

  • register.js — живые подсказки при регистрации (логин, пароль через API).

12. API: ФОРМАТЫ JSON (КРАТКО)
--------------------------------------------------------------------------------

POST /upload/ — multipart/form-data, поле file, заголовок/куки CSRF.
Ответ: { preview_data, columns, uploadedFileId, uploadedFileName } или { error }.

POST /create_chart/ — application/json, CSRF в заголовке X-CSRFToken при необходимости.
Тело см. раздел 6.1. Ответ при успехе: graphData, layout, columns, preview_data, saved.

POST /register/check-username/ — { "username": "..." }
Ответ: { ok, state, message }.

POST /register/validate-password/ — { "password": "..." }
Ответ: { ok, valid, messages }.

POST /api/user-theme/ — { "theme": "light"|"dark" }

GET /api/saved-charts/ — { "charts": [ { id, chartType, fileName, uploadedFileId,
  createdAt, createdAtIso, chartTitle }, ... ] }

GET /api/saved-charts/<id>/ — полный объект настроек + dataJson, layoutJson,
  previewHtml, columns, fileName, fileData (base64), settings, uploadedFileId.

DELETE /api/saved-charts/<id>/ — { "ok": true }

13. БЕЗОПАСНОСТЬ И ДОСТУП К ДАННЫМ
--------------------------------------------------------------------------------

  • CSRF: включён для POST; тесты проверяют отказ без токена при enforce_csrf_checks.
  • Сессии Django для гостей обеспечивают изоляцию uploaded_file_id между браузерами.
  • Сохранённые графики и тема профиля — только для login_required + фильтрация по user.
  • Пароли — хэш Django (в auth_user.password).
  • Хранение файлов в БД увеличивает размер БД; для диплома можно обосновать как
    упрощение развёртывания без отдельного файлового хранилища.

16. ТИПОВЫЕ ПОТОКИ ДЛЯ ГЛАВЫ «ПРОЕКТИРОВАНИЕ»
--------------------------------------------------------------------------------

  • Диаграмма прецедентов: гость (загрузка, просмотр, построение без сохранения);
    пользователь (+ сохранение, коллекция, профиль, тема на сервере); администратор (admin).
  • Диаграмма компонентов: браузер (UI + Plotly.js) ↔ Django (views) ↔ MySQL;
    pandas/plotly на сервере как слой аналитики.
  • ER-диаграмма: User 1—1 Profile; User 1—* UploadedFile; User 1—* SavedChart;
    UploadedFile 1—* SavedChart (по факту использования — много графиков могут ссылаться
    на один файл у одного пользователя).
  • Последовательность: загрузка файла → возврат id → запрос create_chart с id и параметрами
    → возврат JSON traces/layout → отрисовка Plotly → опционально запись SavedChart.
